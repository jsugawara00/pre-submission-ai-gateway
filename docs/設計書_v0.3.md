## 1. プロジェクト概要

申請・申告業務において、AIエージェントが関係書類（インボイス、パッキングリスト、分析資料等）と申請内容を総合的に照合し、転記ミス・資料間矛盾・不審箇所を検出するシステムを開発する。

題材はNACCSの「輸入申告事項登録（業務コードIDA／業務項番5001)」を参考にした疑似申告業務とする。ただし特定システムの再現ではなく、「申請書＋添付書類＋照合」の構造を持つあらゆる業務（行政・金融・保険・法務）に転用可能な汎用リファレンス実装として設計する。

### 開発の狙い

- GitHubポートフォリオの看板プロジェクト。「一人でここまで作れる」と思わせる完成度
- 現行システムの開発者が見て「これはすごい」と思えるプロトタイプ
- 30年の貿易実務とNACCS利用経験に基づく、実務者にしか書けない仕様

## 2. NACCSギャップ分析（本システムの存在理由）

IDA業務仕様書（5001、2026.03版）を分析した結果、NACCSが行うチェックは以下に分類される。

- 形式チェック: 入力欄数、桁数、コード形式
- システム内DB整合性チェック: 品目コードが輸入品目DBに存在するか、担保番号が担保DBに存在し有効期限内か、包括保険番号が保険DBと一致するか等

一方、NACCSが**構造的に行えない**チェックが存在する。

- **入力値と元資料（インボイス、パッキングリスト、B/L、原産地証明書、分析資料等）との一致**
- **元資料群の間の矛盾・相違・不審箇所**

インボイス価格の桁誤り、通貨コードの取り違え、数量の転記ミスは、形式が正しい限りNACCSを素通りする。このため現行実務では「登録→帳票出力→印刷→人間がインボイス等と目視照合→申告送信」という人手工程が残っている。

**本システムは、この「人間の目視照合」工程をAIエージェントで担う。NACCSの再現ではなく、NACCSがやらない側の補完である。**

## 3. 2モード構成と開発意図

本システムは同一の照合エンジンに対して2つの入口（モード）を持つ。各モードはそれぞれ異なる導入アプローチの提案（インスピレーション）として設計されている。

### 3.1 事前モード — 組み込み型（Built-in）

**意図: 「既存の申請システムにAIチェックを融合すると、こうなる」を見せる。**

疑似申告フォームに入力し、関係書類を添付して「事前チェック」を実行すると、登録前に転記ミス・矛盾を検出し、該当フィールドにインラインでエラーを表示して差し戻す。将来、申請システム本体にAIチェックを組み込む際のリファレンスとなるデモ。

フロー: フォーム入力 ＋ 資料添付 → AI照合 → 不備あり: フィールド単位のエラー表示／不備なし: 登録可能

### 3.2 事後モード — 後付け型（Add-on）

**意図: 「既存システムに一切触れなくても、間接的にAIチェックを導入できる」を見せる。**

既存システム（NACCS等）から出力された登録内容の帳票PDFと、元資料一式をまとめてアップロードすると、AIが転記ミスと資料間矛盾を一括チェックしたレポートを返す。基幹システム非侵襲で、現行業務に明日から挿し込める導入パターンのデモ。

フロー: 登録帳票PDF ＋ 元資料一式をドロップ → AI照合 → チェックレポート出力

### 3.3 設計上の核心

- **両モードは同一の照合エンジンを共有する**（エンジンの汎用性の証明）
- **照合結果の表示UIは両モードで完全に同一とする**（入口が違っても同じ品質のチェックが出ることを視覚的に証明する）
- この構成自体が「AIチェックの導入形態は選べる。エンジンは一つでよい」というメッセージになる

## 3.5 聞き返し機能（確認チャット）― 判読不能箇所のヒューマンインザループ

### 背景

貿易実務ではFAX由来の印刷物をスキャンしたPDFなど、判読が困難な書類が日常的に流通している。
AIが不鮮明な文字を「推測」で読むことはハルシネーションの温床であり、照合システムにおいては最も危険な失敗モードとなる。
本機能は「読めない字は推測せず、人間に聞く」を設計原則として実装する。

### 動作フロー

1. 照合実行時、AIは判読確信度が低い文字・数値を推測せず、読めた候補と確信度を添えて`clarifications`（要確認）として返す
2. UI上に確認チャットパネルが起動し、該当箇所の画像切り抜き（PDFの該当領域）とAIの質問を表示する
   - 例:「この欄は『142,500』と読めますが、最後の桁が『6』の可能性もあります。原本を確認して正しい値を入力してください」
3. 人間が値を入力する
4. **AIは人間の回答を鵜呑みにせず、他の数値・文脈と突き合わせて検算してから受理する**
   - 例:「その場合インボイスの明細合計（124,500）と一致しません。通貨はUSDで間違いないですか？」
   - 回答が周辺情報と整合した時点で初めて「確定値」として受理する
5. 確定値で該当項目のみを再照合（軽量パス）し、最終レポートを生成する

### 監査ログとの連動

人間が確認・入力した値はすべて監査ログに記録する。
「この値はAIの読みではなく、誰が・いつ・原本を見て確定した」という経緯が残るため、判読不能箇所の責任の所在が明確になる。

### ハルシネーション三層防御（本システムの設計原則）

| 層 | 仕組み | 対応する失敗モード |
|---|---|---|
| 第1層 | 照合できない項目は`unverified`として明示（推測禁止） | 資料不足の誤魔化し |
| 第2層 | 判読確信度の低い文字は`clarifications`として人間に質問 | 不鮮明文字の誤読 |
| 第3層 | 人間の回答も文脈と検算してから受理（聞き返しループ） | 人間側の入力ミス |

この三層構造はREADMEおよびAboutページに明記し、本システムの信頼性設計の核として提示する。

## 4. UI設計方針（意図を語るUI）

1. **トップ画面 = コンセプト選択画面**: 機能メニューではなく、2枚のカードを並べる。
   - 事前モード: 「組み込み型（Built-in）— 申請システムにAIチェックを融合するとこうなる」
   - 事後モード: 「後付け型（Add-on）— 既存システムに触れずに、帳票からAIチェックを始める」
   - 画面を開いた瞬間に開発意図が読める構造とする
2. **事前モード画面**: 左に疑似申告フォーム、右に添付資料パネル。「事前チェック」ボタンで照合実行。エラーは該当フィールド直下にインライン表示し、登録をブロック
3. **事後モード画面**: ドロップゾーン（登録帳票PDF＋関係資料）のみのミニマル構成。照合後にレポート画面へ遷移
4. **共通レポートUI**: リスクレベル別の色分け（高: 赤／中: 黄／低: グレー）。項目名・帳票側の値・資料側の値・不一致種別・リスク・説明を一覧表示。両モードで完全共通
5. **確認チャットパネル**: 照合結果に要確認（clarifications）が含まれる場合に起動。該当箇所の画像切り抜き＋AIの質問＋入力欄＋会話履歴で構成。解決するとレポートに「人間確認済み」バッジ付きで反映
6. **Aboutページ**: 開発意図（2モードの狙い、NACCSギャップ分析、ハルシネーション三層防御、汎用化構想）を明文化。GitHub閲覧者・面接官への説明を画面内で完結させる
7. 実装前にワイヤーフレーム（画面の紙芝居）を作成する（5画面＋確認チャットパネルは作成済み・確定）

## 5. 疑似申告フォーム項目案（IDA参考・15〜20項目）

実在のIDA入力画面の完全再現はせず、本質を表現する項目に絞る（オブラート方式）。

### 共通部

| # | 項目 | 照合対象の資料 |
|---|---|---|
| 1 | 申告等種別 | — |
| 2 | 輸入者名／コード | インボイス（Buyer/Consignee） |
| 3 | 仕出人（輸出者）名 | インボイス（Seller/Shipper） |
| 4 | B/L番号（AWB番号） | B/L、パッキングリスト |
| 5 | 積載船（機）名 | B/L |
| 6 | 貨物個数 | パッキングリスト、B/L |
| 7 | 貨物重量（グロス） | パッキングリスト |
| 8 | インボイス番号 | インボイス |
| 9 | インボイス価格条件（建値: FOB/CIF等） | インボイス |
| 10 | インボイス通貨コード | インボイス |
| 11 | インボイス価格 | インボイス |
| 12 | 運賃 | B/L、運賃明細 |
| 13 | 保険金額 | 保険料明細 |
| 14 | 原産地コード | インボイス、原産地証明書 |

### 欄部（品目明細・2〜3欄）

| # | 項目 | 照合対象の資料 |
|---|---|---|
| 15 | 品目コード（HSコード） | インボイス品名からの妥当性確認 |
| 16 | 品名 | インボイス、パッキングリスト |
| 17 | 数量・単位 | インボイス、パッキングリスト |
| 18 | 欄ごとの価格 | インボイス明細 |

### 照合エンジンが検出する不一致の例

- 転記ミス: 価格の桁誤り、通貨の取り違え、数量・個数の不一致、B/L番号の誤記
- 資料間矛盾: インボイスとパッキングリストの数量相違、B/Lと申告の船名相違、原産地証明書と申告原産地の不一致
- 不審箇所: 単価×数量と合計金額の不整合、品名とHSコードの乖離、合計重量の不自然さ

## 6. フェーズ計画（v0.2改訂）

| フェーズ | 内容 | 理由 |
|---|---|---|
| Phase 1 | **照合エンジン＋事後モード**。帳票PDF＋資料のアップロード → チェックレポート | UIが最小で済み、エンジン検証に最短で到達できる |
| Phase 1.5 | **聞き返し機能（確認チャット）**。clarifications対応＋画像切り抜き＋検算ループ | エンジンの信頼性の核。事後モードに直結するため早期に実装 |
| Phase 2 | **事前モード**。疑似申告フォーム＋インラインエラー表示。トップのコンセプト選択画面とAboutページもここで整備 | 同一エンジンを使い回すため二度手間なし |
| Phase 3 | メール／複合機（scan to email）からの資料自動取り込み | 入口の多チャンネル化 |
| Phase 4 | NACCS入力フォーマット対応出力、他業界テンプレート（行政・金融・保険・法務）追加 | 汎用化の実証 |

## 7. 技術スタック

| レイヤー | 採用技術 | 理由 |
|---|---|---|
| 言語 | TypeScript | 型安全。学習済みJS知識の延長 |

| ホスティング | Vercel 等（検討） | 個人開発での運用コスト最小化 |

## 8. データベース設計（初期案）

1. **applications（申請）**: 申請ID、モード種別（事前/事後）、申請者、フォーム入力値（JSON、事前モードのみ）、ステータス、作成・更新日時
2. **check_results（照合結果）**: 結果ID、申請ID、書類種別、不一致リスト（JSON）、リスクサマリ、AI判定の生レスポンス参照
3. **audit_logs（監査ログ）**: 誰が・いつ・何を（アップロード／照合／提出／閲覧）行ったか

添付PDFはオブジェクトストレージ（またはローカル暗号化保存）とし、DBにはパスとハッシュのみ保存。
